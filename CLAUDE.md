# trade_council - autonomous FX swing-trading council

You are working in `D:\trade_council\`. This is **v2 of claude_council** - same debate engine, specialized for autonomous trading. Each debate produces a paper_trader plan YAML, the orchestrator schedules follow-up debates per the moderator's `next_review` decision, and a portfolio overseer enforces cross-debate risk limits.

## Relationship to `D:\claude_council\`

`D:\claude_council\` is the general-purpose v1 debate tool - kept for manual research debates and unchanged by this project. Both versions coexist.

`D:\trade_council\` is the autonomous-trading specialization. Key differences from claude_council:

- Specialists: 6 FX swing roster only (the calibration specialists were stripped).
- Moderators: only `fx_swing_chair` (other profiles were stripped).
- Added: `plan_generator` post-debate phase -> paper_trader plan YAML.
- Added: built-in `should_act` / `next_review` / `confidence_level` decisions every debate (Phase 2).
- Added: portfolio overseer with deterministic pre-check + LLM-on-flag (Phase 2).
- Added: autonomous scheduler that fires due debates and paper_trader cycles without user interaction (Phase 3).

## Hard scope rules

- **You may only create or modify files inside `D:\trade_council\`.** Never write to `D:\claude_council\`, `D:\ML_Trend_Predict\`, or any other folder.
- **Paper trading only in v2.0.** No real-broker integration. The `paper_trader` engine simulates fills against ForexFactory calendar + yfinance prices.
- **Specialists never get write tools.** `WebSearch` / `WebFetch` are allowed for specialists whose frontmatter opts in. No other tools.
- **The plan-writer agent gets no tools.** Pure transformation: transcript + decisions -> YAML.
- **API key handling.** When the API backend is wired (Phase 2), `ANTHROPIC_API_KEY` lives in `.env` (gitignored). The Claude Code subprocess backend NEVER inherits this env var so subscription auth works.

## Conventions

- **Plan YAML schema** is canonical at `src/trade_council/plan_generator.py::PLAN_SCHEMA_REFERENCE`. If you change `paper_trader/plan.py` dataclasses, update that reference string too.
- **Trade plans** land in `trade_plans/` named by their `plan_id`. Gitignored.
- **Debate outputs** land in `debates/<date>_<slug>/`. Gitignored.
- **Ledger state** is `paper_trader`'s JSON-backed store, gitignored.
- **No tools to debaters by default.** Each specialist explicitly opts into tools via `tools_allowed:` in frontmatter. Same for the moderator (currently no tools).

## Architecture (current)

```
specialists/        6 FX specialists + _template
moderators/         fx_swing_chair only
overseers/          (Phase 2) portfolio_overseer
src/trade_council/
  ├── debate.py             # run_debate(): research -> debate -> plan_generator
  ├── research.py           # pre-debate WebSearch packet
  ├── specialist.py         # Specialist class
  ├── moderator.py          # Moderator with structured-output parsing
  ├── transcript.py         # turns + decisions + serialization
  ├── plan_generator.py     # post-debate YAML emission (NEW in v2)
  ├── formats/council.py
  ├── backends/             # claude_code (default) + api (stub -> wired in Phase 2)
  └── paper_trader/         # plan loader, calendar, prices, ledger, engine, notify
scripts/
  ├── run_debate.py         # CLI: one-off manual debate
  ├── run_paper_trader.py   # CLI: poll one plan
  └── run_autonomous.py     # (Phase 3) scheduler-driven daemon
trade_plans/        # autogenerated YAMLs, gitignored
debates/            # debate outputs, gitignored
```

## Build sequence (v2)

- **Phase 1 (this commit):** Copy from claude_council, strip non-trading roles, rename package, add `plan_generator`. Manual debates still work end-to-end.
- **Phase 2:** Built-in decisions (`should_act`, `next_review`, `confidence_level`), portfolio overseer integration, fx_swing_chair updates, API backend.
- **Phase 3:** Autonomous scheduler (`run_autonomous.py` + `schedule.sqlite`), cost cap, KILL file.
- **Phase 4:** End-to-end smoke test of the autonomous loop.

## Before making changes

1. Read `STATUS.md` for current phase and what's wired vs. not.
2. Read `README.md` for usage and CLI examples.
3. If touching `paper_trader/plan.py` dataclasses, also update `plan_generator.py::PLAN_SCHEMA_REFERENCE`.
4. If touching `Specialist` / `Moderator` / `Format` public API, run a smoke debate to confirm nothing broke.
